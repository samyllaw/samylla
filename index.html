EULER Team Center

A bespoke internal web application designed to guide new EULER employees through onboarding, help them meet their teammates, and give them branded tools from day one.

HTML · CSS · Vanilla JS Serverless Canvas API localStorage
DeliverableInternal Web App
RoleProduct Designer / Developer
CompanyEULER
DateMarch 2026

The Problem

EULER was scaling rapidly across multiple teams: Design, Development, Business, CX, Sales, Marketing, QA, and Support. New hires were joining frequently, but there was no centralised place to find out what to do, who to talk to, or how to present themselves as part of the EULER brand. Onboarding happened through scattered Slack messages, email threads, and word of mouth.

"New employees were losing their first week to confusion about basics — finding teammates, setting up tools, getting branded assets. We needed to fix the experience before scaling further."

The challenge was to design something that felt polished and trustworthy from the first moment an employee opened it, while being lightweight and easy to maintain without a backend or database.

Feature Set

Rather than build five separate pages, the design intent was to create a single, unified app that a new hire could open and feel they had everything in one place. Navigation was designed to feel like a lightweight SaaS product, not an internal wiki.

Dashboard Home

A personalised landing page showing an "Up Next" widget that surfaces pending tasks, live team stats, and quick-access cards. Completion state persists via localStorage so employees can return to exactly where they left off.

Onboarding Checklist

A two-phase checklist separating universal tasks from team-specific tasks. New hires select their team from a pill selector, revealing their tailored task list. Progress is tracked with an animated progress bar and a live "X of Y done" counter.

Team Directory

A searchable, filterable directory of all EULER employees. Search by name, role, or Slack handle, and filter by team. Each member card expands into a slide-over panel showing contact details. Team badges use distinct colour coding per department.

Email Signature Builder

A live-preview form that generates a fully branded EULER HTML email signature. Users upload their photo, fill in their name, title, email, and LinkedIn, and immediately see an accurate preview. A one-click copy to clipboard delivers the signature ready for Gmail.

Zoom Background Generator

A canvas-based tool that lets employees personalise branded Zoom backgrounds with their name and title overlaid. Four theme options (Pattern, Light, Office, Waves) are available, with a one-click PNG export at full 1920×1080 resolution.

Password Gate

The app is protected by a team-shared password screen, keeping it private to EULER employees while remaining serverless. Features an animated card entrance, shake-on-error validation, and a smooth fade-out unlock transition.

Design Decisions

Every element in the interface was designed with deliberate intent. Below are the key decisions and the reasoning behind them.

Architecture

Single-file, serverless

The entire app is a single HTML file with no external dependencies beyond a Google Font. It can be hosted on any static file server, shared via a link, or run locally. Session persistence uses localStorage, and the password gate uses a client-side hash. This dramatically reduces maintenance overhead for a small team.

Navigation

SPA-style switching without a framework

All five sections live in the same DOM and are shown/hidden with CSS display toggling. Page transitions use CSS animations triggered by JavaScript class swaps, giving a smooth SPA-like feel without React or Vue. This keeps the file size tiny while delivering a polished experience.

Visual Design

Blue as the single accent, restraint elsewhere

EULER's brand is built around a strong, confident blue (#2563EB). The interface uses this as the sole accent colour for active states, CTAs, progress fills, and links. All other colours are neutrals. Team department badges are the only exception, where distinct hues help scanning in the directory.

Motion

Purposeful micro-animations

Every animation serves a functional purpose. The shimmer on the progress bar indicates it is live. The checkbox "pop" animation gives satisfying feedback when a task is completed. Page transitions use a gentle fadeSlideUp to signal context change. None of these animations exceed 400ms, keeping the interface feeling fast.

State

Surgical DOM updates over full re-renders

The "Up Next" widget uses two render strategies: a full render on first load with animated progress bars, and targeted DOM updates on subsequent interactions. Toggling a checkbox updates only the relevant check, tally, and bar width, avoiding animation re-triggers and flicker.

Responsiveness

Three distinct breakpoints

The sidebar collapses completely on mobile, replaced by a fixed hamburger and a slide-in overlay. Two-column grid layouts stack vertically below 680px. Member card metadata is hidden on small screens to reduce clutter. All interactive targets are at least 44px tall on mobile for comfortable touch use.

Brand

Signature and Zoom tools as brand enforcement

Every employee who goes through onboarding leaves with a pixel-perfect EULER email signature and a branded Zoom background, with no design team involvement required. The tools bake in brand defaults and make it impossible to produce an off-brand output.

A Closer Look at Each Module

Onboarding Checklist

Personalised per team, completed at your own pace

The two-step checklist structure reflects the reality that some tasks apply to every new hire — like joining Slack channels — while others are team-specific. Selecting a team updates the checklist in real time, and task state is persisted across sessions so the page always reflects current progress. Completed tasks fade and strike through, giving a satisfying sense of forward momentum.

Team Directory

Find anyone on the team in seconds

The directory solves one of the most common new-hire frustrations: not knowing who does what. Real-time search across name, role, and Slack handle makes finding anyone instant. Team-coloured filter pills use distinct hues per department so visual scanning is fast. Clicking any member card opens a slide-over panel with their contact details without leaving the page.

Email Signature Builder

Brand-consistent signatures, generated in under a minute

Before this tool, new hires would copy signatures from colleagues or skip them entirely, leading to brand inconsistency. The builder renders a live preview that updates as you type. The EULER logo and "Book a Demo" CTA button are baked into every signature by default. Company details (website, meeting links) are locked behind a collapsible warning for non-business teams.

Zoom Background Generator

Branded video presence, personalised on the fly

The Zoom background generator uses the HTML Canvas API to composite a pre-designed EULER branded background image with the employee's name and job title rendered in the exact correct typeface and position. Four visual themes are available: Pattern, Light, Office, and Waves. The output is exported as a 1920×1080 PNG ready to upload directly to Zoom.

Outcomes

8Teams supported with personalised checklists
0Backend servers required to run the app
4Branded Zoom background themes available

Every new employee now leaves their onboarding with a brand-consistent email signature, with no design team involvement and no email chain asking for help. The directory reduces the "who do I even ask?" problem that plagues early days at any startup. The "Up Next" widget ensures no one loses track of what still needs to be done.

The app also demonstrates a design philosophy: internal tools deserve the same level of care and craft as consumer-facing products. The animations, the component polish, the mobile responsiveness — none of these were strictly necessary, but they signal to new employees that EULER is a company that cares about quality end to end.

Reflection

What worked well

The decision to build serverless and single-file was the right call. It removed all deployment complexity and made the tool trivially easy to update — just edit the HTML. The persistent task state via localStorage proved genuinely useful; employees could return days later and pick up where they left off. The two-tier checklist structure naturally scaled as EULER added more teams.

What I'd do differently

If rebuilding, I'd separate the data (team members, task lists) into an editable JSON structure rather than embedding it directly in the JavaScript. This would make it easy for a non-developer to update the directory or add tasks without touching code. I'd also consider adding an admin mode with a simple password for content editing in-app.

Future additions

A "Welcome message" from leadership, perhaps a short video embed on the home screen. The signature builder could be extended with additional layouts. A "Resources" section with quick links to commonly referenced internal documents would also reduce Slack noise around "where do I find X?"

Key principle

Constraints drive creativity. Working without a framework, without a backend, and with a tight scope forced decisions that made the final product better: leaner, faster, and more maintainable. The best tools are often the simplest ones that do exactly what is needed and nothing more.

Gmail Signature Generator

Gmail Signature Generator

An internal web application designed to streamline the creation of professional, branded email signatures — eliminating manual HTML editing and ensuring brand consistency across all team communications.

HTML · CSS · JavaScript Figma Clipboard API Email Design
RoleProduct Designer & Developer
Timeline2 weeks
TeamSolo + CX input
CompanyEULER

The Challenge

Employees struggled with creating Gmail signatures: manual HTML editing led to inconsistent branding, high support ticket volume for signature requests, broken images and formatting issues in email clients, and no standardised process for new hires.

Users needed a simple interface requiring no technical knowledge, one-click copy to Gmail functionality, real-time preview of signature changes, and protection of company-wide settings.

The Solution

I designed a clean, two-column interface that separates user inputs from the live signature preview. The design prioritises clarity and ease of use while maintaining professional aesthetics.

Live Preview System

Real-time signature updates as users type, providing instant visual feedback and reducing errors before copying to Gmail.

Smart Field Organisation

Personal details (name, job title, email, LinkedIn) separated from company details (logo, CTA button, website) with clear visual hierarchy.

Protected Company Settings

Company Details implemented as a collapsible accordion with a prominent warning alert, preventing unauthorised changes to brand elements.

One-Click Copy to Gmail

Custom copy function that preserves HTML formatting when pasting into Gmail, eliminating the need for manual HTML editing entirely.

Technical Implementation

Architecture

Base64 image embedding

Images are converted to base64 data URLs and embedded directly in the signature HTML. This eliminates dependency on external hosting and prevents broken image links across email clients.

Email HTML

Email-optimised table layout

Table-based layout (360px total: 160px left column + 200px right column) ensures consistent rendering across email clients. Inline CSS styles guarantee proper formatting in Gmail, Outlook, and Apple Mail.

UX

Progressive disclosure

Hiding company settings in a collapsed accordion reduced user errors while maintaining access for authorised personnel. Real-time feedback eliminated the trial-and-error cycle of editing, copying, and pasting to test changes.

Validation

Smart URL handling

LinkedIn URL validation automatically adds the https:// protocol when needed, preventing broken links from user input errors without requiring technical knowledge from the user.

Results & Impact

93%Reduction in signature creation time (30 min → 2 min)
0Signature-related support tickets after launch
100%Brand consistency adherence across all signatures

New hires can create signatures independently. The CX team was freed to focus on higher-value tasks. The tool continues to serve the company daily, enabling every new hire and employee update to maintain professional, on-brand email communications with zero friction.

Key Learnings

Progressive disclosure works

Hiding company settings in a collapsed accordion reduced user errors while maintaining access for authorised personnel.

Real-time feedback is essential

Live preview eliminated the trial-and-error cycle of editing, copying, and pasting to test changes in Gmail.

Email clients are unforgiving

Table-based layouts and inline CSS are still necessary for email compatibility. Modern CSS flexbox and grid don't work reliably across email clients.

File size matters

Gmail has strict signature size limits. Proactive guidance on image compression (50 KB max) prevented issues and reduced support questions.

Inclusive Design and Principles for More Accessible Digital Products

Leading and applying accessibility best practices in digital products is meaningful, but can also be challenging. This piece explores key principles that help us think and design more inclusively, grounded in widely adopted accessibility guidelines.

What is inclusive design?

"Exclusion happens when we solve problems using our own biases. We seek out exclusions, and use them as opportunities to create new and better experiences." — Microsoft Inclusive Design

Building inclusive digital solutions means looking at the experience as a whole: who is using the product and how they interact with it. When we think about people, it's important to remember that everyone has limitations, and those limitations can be permanent, temporary, or situational.

Permanent limitations

These may include different types of disabilities, such as cognitive, auditory, mobility, speech, or visual impairments. Age-related changes that happen over time also fall into this category. Each situation comes with different needs and considerations.

Temporary limitations

These affect us for a limited period of time. A sore throat caused by the flu, for example, may make it uncomfortable to speak, turning something as simple as answering a phone call into a challenge.

Situational limitations

These are influenced by external factors. Using a phone outdoors under strong sunlight can make content difficult to see on screen, even with brightness at 100%.

Keeping these scenarios in mind, digital accessibility and inclusive design work together to create better experiences in a realistic and practical way. There is no single best practice that solves every possible use case. Accessibility is achieved through a set of thoughtful decisions that make a product usable by more people, in more situations.

"If we use our own abilities as a baseline, we make things that are easy for some people to use, but difficult for everyone else." — Microsoft Inclusive Design

Building more accessible products

The Web Content Accessibility Guidelines (WCAG) are based on four core principles. Here's a simple overview of each.

01

Perceivable

Information presented in the product should be clear and perceivable to all users. This includes good practices around text readability, color contrast, media usage, and clear visual distinction between interface elements.

02

Operable

All users should be able to operate and interact with the interface and its components. Navigation should support keyboard and assistive technology, allow enough time to complete actions, avoid visual effects that cause discomfort, and not create unnecessary barriers.

03

Understandable

Content and interactions should be easy to understand. Users should always know what is happening in the interface and what is expected of them next, without having to guess.

04

Robust

Content should work across different devices, browsers, and assistive technologies. When creating new components, it's important to clearly define their name, role, and value so users understand what they're interacting with.

In practice, it's not always possible to meet every accessibility requirement throughout a product's lifecycle. There will always be opportunities to improve, revisit decisions, or learn something new before implementing changes. According to Microsoft Inclusive Design, accessibility is an attribute, while inclusive design is a method.

Inclusive design helps us recognize exclusion and understand how to address it, learn from diversity, and think more broadly. Often, a small improvement can positively impact the experience across multiple scenarios. No matter the scale, applying accessibility and inclusion principles leads to better outcomes for users and ultimately results in stronger, more thoughtful digital products.
